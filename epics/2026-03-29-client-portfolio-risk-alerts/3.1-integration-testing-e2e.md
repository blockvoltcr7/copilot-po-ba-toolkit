# 3.1 Integration Testing E2E

## User Story
As a Tech Lead, I want end-to-end integration tests covering the full risk alert pipeline (monitoring → calculation → alerting → notification → audit trail), so that we can deploy with confidence that the entire flow works correctly.

## Acceptance Criteria
- **Given** a test portfolio with known risk drift, **When** the monitoring engine runs, **Then** the integration test verifies an alert is generated, delivered to the dashboard, sent via email, and logged in the audit trail.
- **Given** a test portfolio within tolerance, **When** the monitoring engine runs, **Then** the integration test verifies no alert is generated and no notification is sent.
- **Given** the email service is unavailable, **When** an alert is generated, **Then** the integration test verifies the alert is queued for retry and the dashboard notification is still delivered.

## Edge Cases
- Test data cleanup between runs — each test should create and destroy its own test data
- Parallel test execution — tests should not interfere with each other
- Timing-sensitive assertions — use polling with timeout instead of fixed sleep delays
